## Overview

Replace all direct `SqlParserError<"message">` usage with `FormatError<ErrorID, Args>` to use the error code registry. The FormatError type is already defined in `src/dbtyper-error.ts` and formats error messages as `[dbt:CODE] message`.

**Current State:**

- FormatError type already exists
- 357 error codes in registry
- All error creation sites use `SqlParserError<"raw message">`
- Need to migrate all sites to use `FormatError<ID, Args>`

**Goal:**

- Replace all `SqlParserError<"message">` with `FormatError<ErrorID, Args>`
- Error messages will include `[dbt:CODE]` prefix automatically
- Maintain all existing functionality and tests

## Research Findings Summary

**FormatError Type Structure:**

```typescript
type FormatError<ID extends keyof Errors, Args extends string[]> = Errors[ID] extends {
	code: infer Code
	msg: infer Msg
}
	? Msg extends readonly string[]
		? `[dbt:${Code & (string | number)}] ${JoinWithArgs<Msg, Args>}`
		: never
	: never
```

**Example Usage:**

```typescript
// Old way:
SqlParserError<"Unknown table in FROM">

// New way:
FormatError<"UNKNOWN_TABLE_FROM", []>
// Results in: "[dbt:400] Unknown table in FROM"

// With interpolation:
FormatError<"CLOSING_BRACKET_NOT_FOUND", [")", ""]>
// Results in: "[dbt:108] Closing bracket not found: )"
```
